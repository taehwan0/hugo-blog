#### Fireside — 콘텐츠 중심 B2B 메신저 서비스

`Spring Boot` `Java` `RabbitMQ` `MyBatis` `OCI` `Prometheus` `Grafana`

Fireside는 기업 고객을 위한 B2B 메신저 서비스입니다.
클라우드(멀티테넌시), 단일 설치형, 온프레미스까지 세 가지 환경을 모두 지원해야 했기 때문에, 하나의 코드베이스로 다양한 운영 형태에 대응할 수 있는 서버 구조를 설계하는 것이 핵심 과제였습니다.

멀티테넌시는 스키마 단위로 테넌트를 격리하는 방식을 선택했습니다. DB 인스턴스 분리는 비용이 과했고, Row-level 격리는 MyBatis 환경에서 모든 쿼리에 테넌트 조건을 관리해야 하는 부담이 컸습니다. 무엇보다 단일 설치형과 온프레미스 환경도 동일한 코드로 운영해야 했기 때문에, 코드에 최소한의 영향을 주는 방향이 중요했습니다. 인증 단계에서 테넌트를 확보하고, MyBatis 쿼리 실행 시 자동으로 스키마를 바인딩하는 구조를 만들어 개발자가 멀티테넌시를 별도로 의식하지 않아도 되도록 했습니다.

테넌트 생성도 자동화했습니다. 결제 이벤트를 수신하면 management 서버가 테넌트 스키마와 초기 데이터를 자동으로 구성하는 방식입니다.

**멀티DB 환경에서의 조회 성능 개선**

Fireside는 B2B 서비스 특성상 MySQL, MSSQL, Oracle을 동시에 지원하고 있었습니다. 여기에 레거시 버전 호환까지 유지해야 했기 때문에, DB별로 최적화된 쿼리를 작성하기 어려운 상황이었습니다. MyBatis로 공통 쿼리를 유지하다 보니 인덱스 활용도 제한적이었고, 메시지 테이블에 데이터가 쌓이면서 특정 대화방 조회 시 Full Scan이 발생해 API 응답이 600~800ms까지 느려졌습니다.

먼저 할 수 있는 것부터 했습니다. 조회 빈도가 높은 컬럼 기준으로 인덱스를 설계하고, 조인 연산을 줄이기 위해 테이블 반정규화를 적용했습니다. 이것만으로 응답속도를 50~200ms 수준까지 개선할 수 있었습니다.

하지만 근본적인 문제는 멀티DB 지원이었습니다. 세 종류의 DB를 모두 신경 쓰느라 어느 하나도 제대로 최적화할 수 없는 구조였습니다. 실제 운영 중인 고객사를 조사해보니, MSSQL이나 Oracle을 사용하는 곳이 단 하나도 없었습니다. 이 데이터를 근거로 MySQL 단일화를 제안했고, 처음에는 "미래 고객사를 포기할 수 없다"는 이유로 거절되었지만, 현실적인 운영 데이터를 기반으로 재설득하여 MySQL에 최적화된 운영 전략으로 의사결정을 이끌어냈습니다.

**API 응답속도를 1/4~1/6 수준으로 단축**하고, DB 기술 전략 전환의 기반을 마련했습니다.

**CI/CD 파이프라인 구축 및 컨테이너 전환**

빌드를 한 번 돌리면 20분이 걸렸습니다. 사내 Jenkins는 외부망이 차단된 환경이라, 새 라이브러리가 필요할 때마다 관리자에게 수동으로 요청해야 했습니다. maven으로 war를 빌드하고, scp로 zip 파일을 전달한 뒤, 서버에서 unzip하고 실행 스크립트를 돌리는 방식이었습니다. 팀 전체의 개발 흐름이 이 과정에서 자주 막혔습니다.

먼저 사내에 Nexus를 구축하여 의존성 관리를 자동화했습니다. 특정 outbound만 열어두는 방식으로 보안 정책을 유지하면서도, 라이브러리를 관리자 없이 가져올 수 있게 했습니다. 그 다음 팀 내 유휴 장비였던 iMac을 GitLab Runner로 구성하여 컨테이너 기반 빌드 파이프라인을 도입했습니다.

war 배포에서 컨테이너 방식으로 전환하면서, 서비스를 Docker Compose로 구성했습니다. 모니터링 스택(Prometheus, Grafana, Loki)도 별도 Compose로 분리하여 필요한 시점에 즉시 올릴 수 있도록 했습니다. 배포도 사내에서는 GitLab Registry, 클라우드에서는 OCI Container Registry를 통해 이미지를 관리하는 방식으로 바꿔, 업데이트가 훨씬 간결해졌습니다.

**빌드 시간을 20분에서 8분 이하로 단축**했고, 현재는 FE/BE 빌드를 완전히 분리하여 병렬 빌드를 진행하고 있습니다.

#### DigitalPage — AI 메모 서비스

`Spring Boot` `Java` `MyBatis` `MySQL` `AWS`

DigitalPage는 쉽고 빠르게 기록하는 AI 메모 서비스입니다. 백엔드 개발자로 근무하며 API 개발과 인증 구조 개선 업무를 수행했습니다.

**레거시 인프라 마이그레이션 (Lambda → EC2)**

서비스의 API 서버가 AWS Lambda 위에서 돌아가고 있었는데, Java 런타임 특성상 Cold Start가 심각했습니다. 사용자가 요청을 보낼 때마다 수 초씩 기다려야 하는 상황이 빈번했고, 상시 가동이 필요한 메모 서비스의 특성과 맞지 않았습니다.

Serverless가 모든 상황에 적합한 것은 아니라고 판단하여, EC2 기반의 API 서버로 마이그레이션을 수행했습니다. Cold Start 문제를 근본적으로 해결하며 **평균 API 응답속도를 3배 이상 향상**시켰습니다.

#### DigitalPage Web Clipper — Chrome Extension

`React` `TypeScript` `Zustand`

채용 연계형 인턴으로 입사하여 진행한 프로젝트입니다.

DigitalPage의 모토는 "쉽고 빠른 기록"이었지만, 정작 웹에서 콘텐츠를 바로 메모할 수 있는 경로가 없었습니다. 웹 서핑 중 마음에 드는 글이나 이미지를 저장하려면 앱을 열고, 복사하고, 붙여넣는 과정을 거쳐야 했습니다.

이 문제를 해결하기 위해 Chrome Extension 형태의 Web Clipper 개발을 직접 제안했습니다. 기획, 디자인 협의, React/TypeScript/Zustand 기반 개발, Chrome Web Store 배포까지 전 과정을 전담했습니다. **사내 우수 과제로 선정되어 전사 발표를 진행**했고, 이 프로젝트를 계기로 정규직으로 전환되었습니다.
